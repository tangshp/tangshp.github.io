freeRTOS-03-freeRTOS中的互斥

发表于2022-05-18|更新于2023-02-15|FreeRTOS
|字数总计:1k|阅读时长:3分钟|阅读量:

freeRTOS中的互斥概念

假设StartTask1和StartTask2如下所示,变量a表示两个任务共执行了多少次

uint16_t a = 0;
void StartTask1(void const *argument){
  for (;;)  {
    osDelay(5);
    a++;
  }
}

void StartTask2(void const *argument){
  for (;;)  {
    osDelay(5);
    a++;
  }
}

如下图所示,在汇编层面StartTask1和StartTask2交替的运行,导致调用完两个任务一次后,变量a为1,而不是2

当变量在多个任务中被写时,会出现竞争关系

通过队列实现互斥

如果有多个任务对一个变量写时,可以把写操作放在一个单独的函数(QueueSend)中实现,在该函数中写操作前关闭Tick中断,完成写操作后打开Tick中断;如下图所示:

关闭Tick中断后,代码的运算就是裸机程序

互斥队列支持休眠唤醒

裸机程序中经常有这样一种场景,函数funA中回设置标志位flag,函数funB会根据标志位flag是否是1来执行响应的操作;如以下代码所示:

int flag = 0;
void funA(){
    // maybe set flag to 1
}

void funB(){
    if (flag)
    {
        do_something();
        flag = 0;
    }
}

int main(void){
    while (1){
        funA();
        funB();
    }
}

如果funA中仅有5%的概率将标志位flag设置为1,则循环100次,则有95次浪费时间判断flag是否为1;

如图freeRTOS休眠唤醒所示的休眠唤醒模型就避免了95次浪费时间判断flag是否为1,腾出时间执行StartTask1

唤醒休眠的实现原理

消息队列的核心: 关中断,环形缓冲区,链表
关中断在前面已经进行了说明

链表

数据接受方
当执行读队列的任务StartTask2时,如果队列为空,则在QueueRead中执行以下操作:

  • 使StartTask2休眠,把任务StartTask2放入pxDelayedTaskList
  • 把任务StartTask2插入Queue.list for receive

数据发送方

  • 当执行写队列的任务StartTask1,如果确实往队列中填充了数据,则在QueueSend中执行以下操作:
    QueueSend()
{
    // 关中断
    // 写数据
    // 链表操作
    // 开中断
}
  • 如果队列满了, 任务就会把自己放在Queue.list for send

🌈其中链表操作就是把之前插入Queue.list for receive的任务来唤醒

环形缓冲区

环形缓冲区:通过一个数组arr来完成,并通过writeID和readID来标识当前读写的位置,读写完以后就执行自加操作+取余操作:

  • 最开始时, writeID = readID = 0
  • 写数据时,writeID = (writeID + 1)%sizeof(arr)
  • 读数据时,readID = (readID + 1)%sizeof(arr)

队列具体实现

  • 队列创建:可以使用宏xQueueCreate来创建环形缓冲区, 如下所示创建的环形缓冲区的元素个数为5, 每个元素的大小为sizeof(int32_t)

    #define xQueueCreate( uxQueueLength, uxItemSize ) xQueueGenericCreate( ( uxQueueLength ), ( uxItemSize ), ( queueQUEUE_TYPE_BASE ) )
// 环形缓冲区的创建
xQueue = xQueueCreate(5, sizeof(int32_t));

  • 读队列

    BaseType_t xQueueReceive( QueueHandle_t xQueue, void * const pvBuffer, TickType_t xTicksToWait )

  • 写队列

队列创建

typedef struct QueueDefinition{
    int8_t *pcHead;                
    int8_t *pcTail;                
    int8_t *pcWriteTo; // 指向要写的位置             
    union{
        int8_t *pcReadFrom; // 指向要读的位置         
        UBaseType_t uxRecursiveCallCount;
    } u;
    List_t xTasksWaitingToSend; // 存放要写的任务     
    List_t xTasksWaitingToReceive; // 存放要读的任务 
    volatile UBaseType_t uxMessagesWaiting;// 环形缓冲区中可读元素个数
    UBaseType_t uxLength;       
    UBaseType_t uxItemSize;     
    volatile int8_t cRxLock;
    volatile int8_t cTxLock;
    #if( ( configSUPPORT_STATIC_ALLOCATION == 1 ) && ( configSUPPORT_DYNAMIC_ALLOCATION == 1 ) )
        uint8_t ucStaticallyAllocated; 
    #endif
} xQUEUE;
typedef xQUEUE Queue_t;

队列包括队列头Queue_t和环形缓冲区buffer

  • 队列头部刻画了环形缓冲区buffer的信息, 两个链表
  • 队列头部和环形缓冲区在内存上是连续的
    xQueueSizeInBytes = ( size_t ) ( uxQueueLength * uxItemSize ); 
pxNewQueue = ( Queue_t * ) pvPortMalloc( sizeof( Queue_t ) + xQueueSizeInBytes );

任务中接受数据

任务中接受数据的逻辑如下:

  • 关中断
  • 判断是否有数据
    • 没有数据
      • 可以返回ERR
      • 也可以休眠: (1、把自己添加到队列接受链表 xTasksWaitingToReceive, 以确保发送消息的任务让自己从休眠中恢复; 2、把自己从pxReadyTasksLists中移除, 同时添加到pxDelayedTaskList, 让自己休眠)
    • 有数据
      • 1、拷贝数据, 此时环形队列中有更多的可写节点
      • 2、如果xTasksWaitingToSend中有等待写数据的任务, 则唤醒它
